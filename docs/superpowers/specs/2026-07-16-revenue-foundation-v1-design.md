# 《最后一班：潮汐列车》收益基础设施 v1 设计

## 背景与结论

当前原型已经具备可玩的战斗、签到、首通、每日试炼、军团与三货币系统，但商业化仍停留在单个模拟购买按钮。现有战斗内核不重写；本轮优先补齐可测量、可回滚、不会破坏留存的收益基础设施。

采用“数据驱动的混合变现”方案：保留玩家主动选择的激励广告，增加确定性商品，限制高级货币的可重复免费产出，并让所有购买与广告行为进入统一埋点漏斗。不增加第四种货币，不加入随机付费，不把分享、关注、广告或充值设为签到与成长任务条件。

## 方案比较

### 方案 A：直接增加广告和付费弹窗

开发快，但容易增加首局打断、伤害 D1/D7 留存，也无法判断收入变化来自哪个点位。否决。

### 方案 B：重写战斗内核

投入最大，而且当前没有真实流失数据证明战斗是首要瓶颈。否决。

### 方案 C：收益基础设施优先

保留战斗核心，先修正经济、商店、广告点位和埋点；通过灰度数据决定是否继续改玩法。采用。

## 本轮目标

1. 形成“商店曝光 → 商品点击 → 发起购买 → 购买结果”的完整事件链。
2. 形成“广告入口曝光 → 点击 → 播放结果 → 奖励到账”的完整事件链。
3. 将星票定义为高级货币：普通单局互动不再可重复产出星票，免费星票只保留在签到、首通、活动和里程碑等有上限来源。
4. 用确定性商品目录替换单一模拟按钮，购买结果必须经过“已验证”状态才发货，并保持幂等。
5. 启用两个自然广告点位：奖励三选一重选、普通重复通关结算翻倍。
6. 保留每局一次广告复活和每局一次技能刷新；广告取消或失败不消耗次数、不阻断主流程。
7. 删除“分享成功获得复活”路径；每日试炼成绩卡和军团招募卡继续允许主动分享，但分享本身不发货币或战斗奖励。

## 非目标

- 不接入真实抖音、Apple、Google 或微信支付 SDK。
- 不处理银行卡、收款码、AppSecret、支付密钥或真实订单凭证。
- 不上线月卡、订阅、随机抽取、赛季通行证或付费排行榜。
- 不建立生产服务器；原型继续使用本地存档和 Mock 平台，正式接入边界写入文档。
- 不调整战斗伤害、敌人血量、地图数量或军团玩法。

## 经济规则

### 三货币职责

- 齿轮：高频玩法产出，用于车站升级等常规成长。
- 航线徽记：中频玩法产出，用于地图与长期功能开放。
- 星票：高级货币，只从有限免费来源和确定性商品获得。

现有“点亮潮汐信号”互动奖励从 `1 星票` 改为 `12 齿轮`。同一局重复点击奖励仍保留：打捞信标可领取两次，其他互动各一次。每日试炼继续不结算普通互动资源。

### 确定性商品

首版目录仅放两个一次性商品，避免在服务端订单系统缺失时模拟无限充值：

1. `starter-star-ticket-pack`：展示价 `¥6`，固定获得 `60 星票`，一次性。
2. `abyssal-engine-cosmetic`：展示价 `¥18`，固定获得外观 `深海引擎涂装`，一次性，不增加战力。

商品在购买前完整展示价格、内容和一次性限制。已拥有商品必须禁用购买按钮。重复交易或重复回调不得再次发货。

## 购买架构

### 平台边界

`IStore.purchase(productId)` 返回结构化结果：

- `verified`：平台订单已经由可信服务端确认，包含非空 `transactionId`。
- `cancelled`：玩家取消，不发货。
- `failed`：平台或网络失败，不发货。

Web Mock 可以生成测试交易号，但文案必须标明是模拟购买。生产适配器不得仅凭前端 success 回调发货，必须等服务端验单或平台可信结算回调。

### 领域服务

`ProductCatalog` 只负责不可变商品定义；`PurchaseService` 负责校验商品、验证结果、检查一次性限制和交易幂等，并返回要应用的奖励。界面层只调用服务和提交存档，不自行计算奖励。

### 存档升级

主存档升级为版本 2，新增：

- `purchasedProductIds: string[]`
- `processedTransactionIds: string[]`
- `ownedCosmeticIds: string[]`

读取版本 1 存档时必须无损迁移为版本 2；无效存档继续安全回退默认值。清空本地存档时一并清除所有商业化状态。

## 激励广告规则

### 每局广告点位

- `revive`：失败页每局一次。
- `skill-refresh`：技能耗尽后每局一次。
- `reroll`：普通模式的奖励三选一页面每局一次，完成广告后生成一组不同的确定性候选；每日试炼保持同种子公平，不开放广告重选。
- `double-settlement`：仅普通模式、重复通关胜利结算可用一次；额外发放 `80 齿轮 + 2 航线徽记`。首通、失败、每日试炼不可翻倍。

所有点位都必须显示明确奖励；广告取消或失败后按钮可重试，奖励不得发放。相同结算页或相同奖励页完成后不得重复领取。

### 分享调整

失败页只保留广告复活和放弃结算。`RecoverySystem` 只维护广告复活状态。每日试炼成绩卡和军团招募分享保持纯传播能力，取消或失败不改变游戏资产。

## 埋点与 KPI

新增事件：

- `store_viewed`
- `product_clicked`
- `purchase_started`
- `purchase_result`
- `rewarded_ad_offer_shown`
- `rewarded_ad_clicked`
- `rewarded_ad_result`
- `economy_reward_granted`

事件不记录支付密钥、银行卡、收款码、玩家输入的礼包码原文或其他敏感信息。购买事件只记录商品 ID、结果、奖励摘要和脱敏测试交易 ID；生产环境的完整订单号由服务端安全保存。

核心监控：

- 商店点击率 = 商品点击用户 / 商店曝光用户。
- 购买转化率 = 已验证购买用户 / 发起购买用户。
- 广告完成率 = 完成广告次数 / 点击广告次数。
- 广告奖励到账率 = 奖励到账次数 / 完成广告次数，目标必须为 100%。
- ARPDAU、D1、D7 由正式平台与服务端数据计算；原型只验证事件完整性。

守护指标：广告失败不阻断率 100%；免费与付费玩家都能完成核心内容；付费商品不直接提高基础战斗伤害；星票免费产出有明确周期上限。

## Web 界面

车站新增“航线补给站”商品卡：名称、价格、确定内容、用途、一次性状态和模拟购买按钮。商品卡在移动端单列显示，不出现横向滚动。

奖励页面在三选一下面显示“看广告重选一次”；普通重复通关结算显示“看广告领取同额普通奖励”。按钮在请求中禁用，完成后改为已领取或已使用。

## Cocos 接入骨架

新增 `CommerceController`，只负责把商品点击和广告请求转成场景事件，不包含平台密钥或支付实现。抖音、微信、iOS 和 Android 适配器以后监听这些事件并调用各自 SDK。

## 错误与回退

- 未知商品：拒绝发货并显示配置错误。
- 取消或失败：不修改资产与所有权，可重新操作。
- 重复交易：返回幂等结果，不二次发货。
- 一次性商品重复购买：拒绝并显示已拥有。
- 本地旧存档：自动迁移；字段类型异常：回退默认存档。
- 广告失败：保留点位次数和原流程入口。

## 测试与验收

1. 单元测试覆盖商品目录、已验证发货、取消、失败、未知商品、重复交易和一次性购买。
2. 存档测试覆盖版本 1 到版本 2 无损迁移与新增数组深拷贝。
3. Recovery 测试只保留广告复活，确认每局最多一次。
4. Web 视图测试覆盖商品内容、已拥有状态、广告重选与结算翻倍按钮。
5. Telemetry 测试覆盖新增商业化事件且不包含敏感字段。
6. 全量运行测试、类型检查、生产构建和 `git diff --check`。
7. 浏览器手工验证购买、刷新持久化、星票不再从普通互动产出、广告失败兜底、重选、重复通关翻倍及窄屏布局。

## 上线边界

本轮完成的是可玩、可测的商业化原型，不代表真实收入已经开通。正式上线前仍需开发者主体、平台商户、服务端验单、资产账本、退款补单、实名与防沉迷、隐私政策、税务与银行结算配置，并按目标平台最新规则审核。
