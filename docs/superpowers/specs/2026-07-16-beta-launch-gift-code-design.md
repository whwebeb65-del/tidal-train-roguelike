# 内测资格、开服礼与礼包码设计规格

## 目标

在《最后一班：潮汐列车》的车站中加入一个可试玩的“开服预约中心”，让玩家完成以下闭环：

1. 申请限量内测资格并获得明确的资格状态。
2. 内测玩家领取一次额外先行者补给。
3. 开服活动期玩家领取一次高价值开服礼。
4. 输入礼包码，得到成功、无效、重复、未开放或过期的确定反馈。
5. 所有奖励与资格状态可持久化，刷新页面后不会重复领取。

本轮验证单账号体验和规则边界，不伪装成真实全服名额系统。

## 方案选择

### 方案 A：立即建设完整活动后端

真实账号、全服名额池、码库存和运营后台都由服务端管理，最接近正式上线，但当前项目尚未接入登录、数据库和部署环境，范围过大。

### 方案 B：可移植规则层 + 本地活动中心 + 服务端事件桥接

将资格、领奖、礼包码核销和防重写成无引擎依赖的纯规则；Web MVP 使用独立本地存档演示；Cocos 只提交请求事件；文档明确正式服的服务端权威边界。

采用此方案。它能立即验证体验，又不会把本地演示逻辑误认为正式安全方案。

### 方案 C：静态活动展示页

只展示内测和开服礼包文案，不维护状态。开发最快，但无法测试重复领取、过期和奖励到账，不能形成玩法闭环。

## 活动结构

### 内测资格

- 展示“限量 3,000 席”，但不展示伪造的实时剩余人数。
- 原型中点击“申请内测资格”立即获得单账号模拟资格。
- 同一活动周期重复申请返回“已经获得资格”，不会生成第二份奖励。
- 资格本身不直接改变战斗数值，只开放一次“潮汐先行者补给”。
- 先行者补给：60 齿轮、2 航线徽记、1 星票、非战力徽章“潮汐先行者”。

### 开服列车长礼

- 开服活动期内所有玩家可领取一次。
- 奖励：188 齿轮、6 航线徽记、3 星票、非战力徽章“开服列车长”。
- 设计目的：普通开服玩家至少能立即完成一次车站升级；内测玩家叠加先行者补给后可更快体验下一阶段内容。
- 资格礼、开服礼和礼包码彼此独立，领取其中一项不会阻断其他项。

### 礼包码

原型提供两个可核验活动码：

| 输入码 | 活动 | 奖励 | 状态 |
| --- | --- | --- | --- |
| `TIDE2026` | 潮汐启航 | 66 齿轮、2 航线徽记 | 当前可用 |
| `FIRSTTRAIN` | 首班列车 | 88 齿轮、1 星票 | 当前可用 |

- 输入先去除首尾空格并转换为大写。
- 空字符串、未知码、重复码、未开放码和过期码返回不同原因。
- 单个账号对同一礼包码只能成功一次。
- 原型可以在客户端包含活动码目录；正式服不能依赖客户端隐藏码或决定奖励。

## 状态与接口

新增独立的 `LaunchCampaignState`，不升级 `PlayerSave` 版本：

```ts
interface LaunchCampaignState {
  readonly version: 1;
  readonly campaignId: 'launch-2026';
  readonly betaQualified: boolean;
  readonly betaGiftClaimed: boolean;
  readonly launchGiftClaimed: boolean;
  readonly redeemedCodeIds: readonly string[];
  readonly cosmeticBadgeIds: readonly string[];
}
```

规则层公开：

- `createLaunchCampaignState()`
- `normalizeLaunchCampaignState(candidate)`
- `applyForBeta(state)`
- `claimBetaGift(state)`
- `claimLaunchGift(state)`
- `redeemGiftCode(state, code, nowMs, catalog?)`

所有成功结果返回新的不可变状态与 `CampaignReward`；失败结果返回零奖励和原因，不修改输入状态。
`catalog` 默认使用原型公开码目录；测试和未来服务端适配可以注入只读目录，以验证不同活动窗口而不依赖真实系统时间。

## 时间边界

为保证原型长期可试玩，本地活动目录以明确配置驱动：

- 内测资格申请、先行者补给和开服礼在原型活动 `launch-2026` 中保持开放。
- 礼包码定义仍包含 `startsAtMs` 与 `expiresAtMs`，测试使用固定时间验证未开放和过期分支。
- Web 页面明确显示“原型活动开放中”。
- 正式上线时由服务端下发服务器时间和活动配置，客户端时间不参与最终判定。

## Web 体验

车站在地图区域与军团区域之间新增“首班预约中心”：

- 顶部显示 `FOUNDERS / 3,000`、活动状态和资格标签。
- 两张礼包卡分别承载内测资格/先行者礼与开服列车长礼。
- 礼包码输入框支持回车和按钮兑换，展示可测试公开码 `TIDE2026`。
- 成功领取后，顶部三货币立即更新；对应按钮改为不可重复状态。
- 徽章只作为身份纪念展示，不提供伤害、生命或掉落加成。
- 390px 宽度下所有活动卡和兑换区改为单列，不产生横向滚动。

## 持久化与奖励提交

- Web 使用独立键 `tidal-train-launch-campaign-v1` 保存活动状态。
- 成功领奖时先计算规则结果，再一次性写入活动状态和 `PlayerSave` 三货币。
- 本地原型无法提供真正事务；正式服必须以一次服务端事务同时写资格、核销记录和资产流水。
- 清空本地存档同时清除玩家、军团和开服活动三个存档键。

## Cocos 与正式服边界

新增请求型控制器事件：

- `beta-application-requested`
- `beta-gift-claim-requested`
- `launch-gift-claim-requested`
- `gift-code-redeem-requested`

Cocos 控制器不自行发奖。正式服 API 必须校验登录账号、活动窗口、全服名额、单账号领取记录、礼包码状态、渠道和库存，并返回服务端资产余额与核销流水号。

## 埋点

新增：

- `beta_application_result`
- `campaign_reward_claimed`
- `gift_code_redeem_result`

礼包码失败埋点只记录活动码 ID 或标准化失败原因；未知用户输入不得原样上报，避免收集无关文本。

## 测试与验收

规则测试覆盖：

- 首次申请成功、重复申请幂等。
- 未获得资格不能领先行者礼。
- 先行者礼和开服礼各自只能领取一次。
- 礼包码忽略首尾空格和大小写。
- 未知、重复、未开放和过期码返回准确原因。
- 畸形本地状态能被安全归一化，非法徽章和礼包记录不会进入状态。
- 所有失败分支都返回零奖励。

集成验收覆盖：

- Web 申请资格、领取两类礼包、兑换 `TIDE2026` 后三货币按预期累计。
- 刷新后按钮保持已领取，重复操作不再加资产。
- 手机单列布局、无横向溢出、控制台无错误。
- `npm test`、`npm run typecheck`、`npm run build` 全部通过。

## 不在本轮范围

- 真实邀请码裂变返利。
- 付费礼包码生成或用户间转卖。
- 实时全服剩余名额、排队和候补系统。
- 运营后台、渠道批次导入和客服补单。
- 基于内测身份提供永久战力优势。
