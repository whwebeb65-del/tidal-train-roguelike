# 《最后一班：潮汐列车》性能与可玩性增强规格

- 日期：2026-07-15
- 项目：project-002-tidal-train-roguelike
- 状态：自主增强实施中

## 1. 本轮目标

在不引入重型 3D 引擎依赖的前提下，把当前可玩原型提升为更接近上线前测试的版本：战斗有更多决策，构筑反馈更清楚，画面有轻量 2.5D 的“列车模型感”，移动端保持稳定的事件驱动渲染。

本轮不接入真实抖音 SDK、不改变已确认的广告复活和分享复活规则，也不加入付费随机抽取。

## 2. 方案选择

### 方案 A：直接补完整 3D 场景

表现上限最高，但依赖 Cocos Creator、模型资源、压缩和真机性能调试；当前没有可用的官方编辑器环境，无法安全交付可验证的 3D 包。

### 方案 B：轻量 2.5D 表现加纯规则战斗层（采用）

用 CSS/Canvas 友好的层次、光晕、透视、潮汐线和车厢结构增强“建模感”，战斗决策全部落到纯 TypeScript 规则层；Web MVP 可以立即验证，Cocos 只需接同一套事件接口。优点是可测试、低包体、低运行时开销，适合抖音小游戏首发。

### 方案 C：只扩充数值和内容

风险最低，但不能解决玩家觉得操作单一和画面表现薄的问题。

## 3. 战斗增强

新增 `CombatLoopSystem`，作为普通战斗和 Boss 共用的纯规则层：

- 自动开炮：造成基础伤害并积累“潮汐动能”；连续行动提升连击，产生少量伤害加成。
- 技能：沿用现有充能和广告刷新规则，造成更高伤害并快速积累动能。
- 维修车厢：每个战斗节点一次，恢复生命但打断连击，给玩家一个“保命还是抢节奏”的选择。
- 潮汐爆发：动能达到 100 后可使用一次，造成高额伤害并清空动能。
- 每个节点生成一个确定性的潮汐修正词缀：平静水域偏维修、急流冲刺偏普通攻击、回声浓雾偏技能；同一种子始终得到同一结果。

规则层只输出新状态和动作结果，Web/Cocos 不直接修改敌人生命或奖励，避免两个客户端产生不同规则。

## 4. 性能与渲染

- 保持事件驱动：没有持续的业务 `requestAnimationFrame` 循环，战斗状态变化时才重绘 UI。
- 战斗画面使用 CSS 层和伪元素完成水纹、阴影、车厢透视和敌人呼吸效果，限制 DOM 数量。
- 给战斗板、车站场景和卡片增加绘制隔离与移动端降级；移动端关闭高成本模糊和非必要动画。
- 增加 `prefers-reduced-motion` 兼容，弱设备和用户主动减少动画时不影响操作。
- 保留性能预算检查：构建后检查包体、模块数量和关键交互不产生控制台错误。

## 5. 验证范围

- 新增战斗规则单元测试：动能、连击、维修上限、爆发门槛、词缀差异、敌人击败和非法操作。
- Web 回归：普通战斗中完成攻击、技能、维修、爆发；失败页复活链路继续可用；新节点重置维修和动能。
- Cocos 回归：HUD 能发出四种战斗动作和战斗状态更新事件。
- 运行 `npm test`、`npm run typecheck`、`npm run build`、`git diff --check`。

## 6. 交付边界

本轮交付的是可验证的性能友好原型，不宣称已经达到正式真机 60 FPS。正式发布前仍需在抖音开发者工具和至少两档安卓设备上进行真机帧率、内存、首屏和广告回调测试。
