# 《最后一班：潮汐列车》列车动态系统设计

- 日期：2026-07-17
- 状态：用户已确认设计方向
- 适用版本：网页公开测试版下一次动态表现升级

## 1. 背景与问题

当前车站中的泡泡列车只有约 4 像素的上下漂浮，战斗中的列车则以一张固定整图显示。炮塔、列车长和伙伴具有局部动作，但车体、航道、水流和动力反馈没有共用速度状态，玩家容易感觉角色站在静态图片上，而不是乘坐一列持续前进的海洋列车。

本次升级保留已经确认的现代海洋 Q 版美术，不重做列车主体，不使用大 GIF，也不引入 3D 引擎。通过程序化分层动画和少量局部光效，让玩家在不操作时也能明确感受到列车正在行驶，并让开炮、受击、技能、Boss 和结算与列车运动建立一致反馈。

## 2. 目标与非目标

### 2.1 目标

1. 覆盖车站待机、出发、战斗巡航、Boss 加速、胜利减速和失败停机的完整行程。
2. 让列车、列车长和伙伴共享同一基础姿态，避免人物与车体脱离。
3. 通过航道后掠、尾流和视差建立持续前进感，不依赖夸张的车体弹跳。
4. 让现有战斗事件驱动列车反馈，但不改变伤害、难度、奖励或结算。
5. 保持四种目标手机尺寸、低画质和减少动态效果模式可用。

### 2.2 非目标

- 不把海洋悬浮列车改成传统轮轨列车，不增加不符合现有造型的车轮。
- 不制作整段视频、GIF、序列帧列车或 3D 模型。
- 不修改现有战斗数值、永久成长、商业化、存档和平台接口。
- 不在本阶段增加列车驾驶操作、路线分叉或速度数值玩法。
- 不重新生成已确认的列车、角色和伙伴主体美术。

## 3. 整体体验节奏

### 3.1 车站待机

车站列车保持低幅悬浮，动力核心和舷窗出现缓慢流光，车尾附近有少量气泡与短尾流。背景维持现有缓慢漂移，但列车与背景使用不同周期，形成轻微视差。待机动作不表现高速行驶，只传达动力系统已经启动。

列车长和伙伴继续使用各自的呼吸、检修与漂浮动作，不强制跟随列车的全部局部变化；它们只叠加列车的基础位移，保持脚部或站位锚点稳定。

### 3.2 点击出发

点击“驾驶泡泡列车出发”后执行以下流程：

1. 立即解锁音频并开始关键战斗资源准备。
2. 列车进入蓄能状态，动力核心在 400 毫秒内变亮，车体轻微后沉。
3. 若关键资源尚未就绪，保持低幅蓄能循环并继续显示装载状态，不提前进入空白战场。
4. 资源就绪后播放约 700 毫秒的驶出动作：尾流增强、列车向航道方向推进、背景产生分层视差。
5. 在水花与亮度遮罩覆盖下挂载战斗场景，避免硬切。

驶出动画只承担表现作用。重复点击、资源失败或场景销毁不得重复创建战斗或结算。

### 3.3 战斗巡航

战斗以列车朝屏幕上方前进的透视关系表现。列车主体仍位于画面底部，环境元素向屏幕下方移动：

- 三条水轨增加按透视缩放的泡沫标记和高光段。
- 近处标记移动更快、尺寸更大，远处标记移动更慢、尺寸更小。
- 列车两侧和尾部显示短尾流，强度随速度变化。
- 背景只做低速视差，避免与怪物航道竞争视觉注意力。
- 车体进行低幅上下起伏、横向惯性和滚转，列车长与伙伴共享该基础姿态。

普通巡航的车体位移上限为横向 2.2 逻辑像素、纵向 2.8 逻辑像素，滚转上限为 0.55 度。周期使用两个不同频率的正弦分量，避免机械重复，同时保持固定输入下结果可预测。

### 3.4 战斗状态变化

使用归一化视觉速度驱动环境后掠和尾流：

| 状态 | 目标视觉速度 | 表现 |
|---|---:|---|
| 车站待机 | 0.18 | 低幅悬浮、慢流光、少量气泡 |
| 出发加速 | 0 → 1.00 | 蓄能后平滑加速 |
| 普通波次 | 1.00 | 稳定巡航 |
| 精英阶段 | 1.08 | 尾流略增强、动力核心变亮 |
| Boss 阶段 | 1.22 | 航道后掠加快、尾流变长、低频动力音略升高 |
| 胜利 | 1.00 → 0.25 | 约 1.4 秒平滑减速，保持动力未完全关闭 |
| 失败 | 1.00 → 0 | 约 0.9 秒失速，动力灯熄灭并缓慢停下 |

速度值只控制视觉和音频表现，不进入战斗规则。

### 3.5 瞬时反馈

- 普通主炮：炮塔后坐约 4 像素、100 毫秒；车体最多叠加 0.6 像素的短促反作用力。
- 潮汐齐射：车体短暂后沉，侧向尾流增强约 260 毫秒。
- 泡泡屏障：列车姿态不突变，护盾展开时动力流光变为海沫青色。
- 极潮爆发：列车先下沉再向前冲，视觉速度短暂增加，但不改变实体坐标或敌人移动。
- 列车受击：按攻击方向施加一次横向位移和滚转冲量，在约 220 毫秒内衰减归零。
- 低生命：生命低于 30% 时动力灯亮度降低并间歇闪烁，尾流减少，不使用持续全屏红闪。
- Boss 出场：在现有 6 秒演出中平滑提升视觉速度，不瞬间跳变。

所有冲量有明确上限并使用时间衰减。多个事件同时发生时先合并再限幅，防止高射速或连续受击造成眩晕。

## 4. 视觉分层

战斗绘制顺序调整为：

```text
远海背景
→ 低速背景气泡
→ 航道与透视泡沫标记
→ 列车尾流和近景水花
→ 掉落物、敌人、炮弹
→ 列车车体
→ 炮塔、列车长与伙伴
→ 护盾、动力核心和舷窗流光
→ 命中、技能和镜头前景特效
→ HUD 与演出遮罩
```

列车继续复用 `bubble-train.webp`。动力核心、舷窗流光、水轨标记和尾流优先使用 Canvas 图元与混合模式绘制。只有程序化图元无法达到现有材质质量时，才允许补充单张透明 WebP 光效；新增首屏资源总量不得超过 80 KB，并必须纳入资源预算脚本。

## 5. 系统结构

### 5.1 新增边界

新增以下独立模块：

- `web/battle/TrainMotionTypes.ts`：阶段、输入、冲量和输出姿态类型。
- `web/battle/TrainMotionController.ts`：速度曲线、基础摇摆、事件冲量、低生命状态和减少动态效果规则。

现有模块的职责变化：

- `BattleScene`：把引擎事件转成列车运动冲量，使用战斗模拟时钟推进控制器，并在暂停、恢复和销毁时同步生命周期。
- `BattleRenderer`：读取列车姿态，绘制航道后掠、尾流、车体、乘员和动力光效；不自行推导战斗状态。
- `BattleRenderInput`：增加只读列车运动帧视图。
- `StationHeroView` 与车站样式：增加待机、蓄能和驶出状态所需的分层元素和 CSS 状态类。
- `AudioManager`：维护一个可复用的低音量动力声层，以参数缓动响应视觉速度；不得每帧创建音频节点。

### 5.2 控制器输入

`TrainMotionController` 每个固定更新接收：

- 当前表现时钟与增量时间。
- 当前阶段和目标视觉速度。
- 生命比例、护盾状态和能量比例。
- 本帧的开炮、受击、技能与结算冲量。
- 画质等级与减少动态效果设置。
- 当前是否暂停或页面隐藏。

### 5.3 控制器输出

控制器复用一份稳定的只读帧对象，包含：

- 基础车体 `x`、`y`、`rotation` 和轻微缩放。
- 炮塔后坐量。
- 视觉速度、尾流强度和航道偏移。
- 动力核心亮度、流光相位和低生命闪烁强度。
- 受击方向、技能推进量和结算减速进度。

渲染器不得缓存下一固定更新后仍会变化的内部引用。

### 5.4 数据流

```text
BattleEngine 固定步长更新
→ BattleEvent 分发
→ BattleScene 映射运动冲量
→ TrainMotionController 更新稳定帧视图
→ BattleRenderer 绘制列车与环境
→ AudioManager 缓动动力声参数
```

列车运动控制器不能反向调用 `BattleEngine`，也不能修改敌人位置、战斗时钟、伤害、生命或奖励。表现故障不会影响模拟和结算。

## 6. 音频联动

动力声保持低于主炮、命中和背景音乐，不持续抢占听觉注意力：

- 车站待机使用很轻的水下动力嗡鸣。
- 出发时增加一次短促蓄能和水流推进声。
- 普通战斗维持稳定音高，精英和 Boss 只做小幅平滑提升。
- 胜利减速时音高与音量缓慢下降；失败时更快衰减至静音。
- 音乐关闭不强制关闭动力声；动力声归入音效开关。

参数变化使用音频自动化或节流更新，不在每个渲染帧重建节点。音频初始化失败时跳过该层，列车视觉动态继续工作。

## 7. 暂停、异常与降级

- 页面隐藏、升级选择、复活层和手动暂停期间，列车表现时钟与战斗时钟一起停止。
- 页面恢复后不补算隐藏期间的运动，不出现突然加速或追帧。
- 车站出发过程中若场景被销毁，取消待处理动画和音频，不继续导航。
- 列车图片加载失败时使用现有安全替代轮廓，并继续绘制航道、尾流和动力核心。
- 音频不可用时只关闭动力声，不阻断出发或战斗。
- 低画质减少背景气泡、尾流段和舷窗流光，不移除航道后掠、车体惯性与状态反馈。
- 开启减少动态效果时，车体摇摆、冲刺、受击偏移和镜头叠加均归零；保留低速航道流光、护盾变化、动力亮度和结算状态，确保信息仍可理解。

## 8. 性能预算

- 控制器固定更新不得创建临时数组、对象或闭包；姿态对象在实例生命周期内复用。
- 航道标记使用固定数量和取模位置，不创建真实粒子实体。
- 新增列车相关 Canvas 绘制命令上限：高画质 24、中画质 14、低画质 8。
- 低画质必须保留每条航道至少一个可见后掠标记和列车两侧基础尾流。
- 不提高现有敌人、炮弹和伤害数字预算。
- 新增可选光效资源必须为本地 WebP，总量不超过 80 KB，且不得阻塞关键战斗资源失败回退。
- 连续两局后，列车控制器、音频节点、事件监听器和动画回调数量必须回到单局初始水平。

## 9. 测试设计

### 9.1 单元测试

- 相同阶段、时间和冲量序列产生一致姿态。
- 待机、出发、普通、精英、Boss、胜利和失败的速度曲线达到规定目标并连续过渡。
- 开炮、受击和技能冲量在规定时间内衰减，不留下永久位置或旋转偏差。
- 多个冲量合并后不超过位移和旋转上限。
- 低生命、护盾和能量只改变表现字段，不改变战斗状态。
- 暂停更新不会推进相位或冲量衰减。
- 减少动态效果输出零车体摇摆、零冲刺和零受击偏移。
- 低画质只降低视觉预算，不改变姿态阶段或速度曲线。

### 9.2 渲染测试

- 列车、列车长和伙伴使用同一基础姿态偏移。
- 炮塔瞄准角和后坐正确叠加在车体姿态上。
- 航道标记按视觉速度移动并在越界后确定性循环。
- 尾流、核心和流光的绘制数量遵守各画质预算。
- 列车素材缺失时安全替代图形仍具有动态姿态。

### 9.3 浏览器回归

在 360 × 800、390 × 844、412 × 915 和 430 × 932 下验证：

1. 车站待机 1 秒内可见列车动力和水流动态。
2. 点击出发后完整播放蓄能、驶出和战斗衔接，不重复启动。
3. 战斗不操作时仍能清晰判断列车正在前进。
4. 普通开炮、主动技能、受击、精英、Boss、胜利和失败分别触发对应动态。
5. 人物和伙伴不脱离列车，HUD 不随车体摇摆。
6. 暂停或切后台后运动停止，恢复时无跳帧追赶。
7. 开启减少动态效果后仍保留必要速度和状态提示。
8. 连续完成两局后无残留动画、音频节点、监听器或未捕获异常。
9. 普通公开 URL 不暴露测试控制接口。

## 10. 验收标准

功能完成必须同时满足：

- 玩家不操作时，在进入车站或战斗后的 1 秒内能够通过航道后掠、尾流或动力流光判断列车状态。
- 车站待机、出发、巡航、Boss、胜利和失败形成连续、无硬跳的速度变化。
- 开炮、受击和技能有列车级反馈，但普通巡航不超过 2.2 像素横移、2.8 像素纵移和 0.55 度滚转。
- 战斗规则、首通奖励、重复奖励和结算结果与升级前一致。
- 四种移动端尺寸、自动低画质和减少动态效果模式均通过回归。
- `npm test`、`npm run typecheck`、`npm run check:assets`、`npm run build`、浏览器冒烟、`npm audit --audit-level=high` 和 `git diff --check` 全部通过后才允许重新部署。

## 11. 实施边界

本设计作为一个独立视觉系统实施，顺序为：纯运动控制器与测试、战斗渲染联动、车站出发转场、音频联动、移动端与生命周期回归、部署。若实现过程中发现现有绘制接口无法在不破坏其他实体的情况下共享列车姿态，只允许增加局部坐标变换辅助函数，不进行无关的渲染器重构。
