# 《最后一班：潮汐列车》原型试玩脚本

## 目的

验证玩家能否在短时间内理解“出发—战斗—选择—互动—首通—车站升级”的核心闭环，并确认重复点击奖励、三货币和地图开放不会造成明显困惑。

## 准备

1. 在项目根目录运行 `npm run dev`。
2. 打开 `http://127.0.0.1:4173/`。
3. 点击底部“清空本地存档”，确保从 1 级车站、漂流郊区开始。
4. 如需移动端观察，将浏览器窗口缩窄到约 390px 宽。

## 主流程

1. 记录从页面打开到第一次有效点击的时间；目标是 30 秒内点击“驶向漂流郊区”。
2. 进入战斗后，观察玩家是否能找到三条航道、自动开炮、汽笛共鸣、维修车厢和潮汐爆发按钮；先连续开炮，确认动能和连击会上升。
3. 在“同局互动奖励”中连续点击“打捞废弃信标”两次，确认齿轮从 0 变为 16，按钮显示 `2/2` 且禁用。
4. 点击“给漂流者递水”和“点亮潮汐信号”，确认航线徽记和星票分别增加，且每项只能领取一次。
5. 先用“维修车厢”恢复一次生命，再用技能或继续开炮击败当前敌人；确认维修按钮本节点只可使用一次，选择一个乘客或模块后进入路线选择。
6. 依次选择路线、奖励和下一条路线，观察风险、节点类型和构筑提示是否足够清楚。
7. 进入 Boss，观察节点修饰标签；连续攻击积满 100 动能后点击“潮汐爆发”，确认 Boss 击破并首通结算显示：400 齿轮、10 航线徽记、3 星票。
8. 回到车站，连续升级两次，确认车站达到 3 级且“旧港口”从锁定变为开放。
9. 点击“模拟购买 60 星票”，确认商店边界只在 Mock 中生效，并观察星票变化。

## 重点观察

- 第一次有效点击是否明显；
- 互动按钮的“本局次数”和“已领完”是否容易理解；
- 三种货币是否能区分用途；
- 首通奖励与重复通关奖励是否清楚；
- 地图锁定条件（车站等级）是否可预测；
- 移动端窄屏下按钮是否仍可点击；
- 玩家是否愿意在首通后再次挑战不同构筑。
- 节点修饰是否能让玩家理解不同的攻击、技能与维修选择；新节点是否重置维修、爆发、连击和动能。
- 窄屏设备上战斗状态栏和四个动作按钮是否仍然清晰、可点击。

## 通过标准

- 试玩者无需口头解释即可完成第一次出发、一次互动、一次奖励选择；
- 重复点击不会重复发放，达到上限后有明确反馈；
- 首通奖励在结算页和资源栏中一致；
- 车站升级与地图开放状态刷新正确；
- 浏览器控制台无阻断性错误。
- 普通战斗不应产生持续计时器或持续 `requestAnimationFrame` 业务循环；每次动作只触发一次状态更新和一次 UI 重绘。
- 性能观察以“无长任务、无明显掉帧、动画可在窄屏关闭”为目标；本脚本不替代抖音目标机型的真机 60 FPS 验证。

## 复活与技能刷新回归路径

1. 清空本地存档，开始一局普通战斗，连续点击“模拟受击 · -35”至失败。
2. 点击“看广告复活”，确认生命恢复到 60/100，并回到当前战斗。
3. 再次受击至失败，点击“分享战绩复活”，确认生命恢复到 50/100，并回到当前战斗。
4. 再次受击至失败，确认广告复活和分享复活都显示“已用”，点击放弃并确认进入 `RUN SETTLED` 结算页。
5. 开始新局，先释放一次技能，确认技能充能变为 0/1 且出现“看广告刷新技能”。
6. 点击“看广告刷新技能”，确认技能充能恢复为 1/1；再次检查本局刷新入口不可重复使用。

## 观察记录

- 取消广告或分享、平台回调失败时，不应消耗对应次数或技能刷新次数；
- 复活成功后应保留当前敌人生命、路线和同局互动奖励；
- 分享卡应包含地图、深度、乘客、模块、失败原因和明确 CTA；
- 快速重复点击复活或刷新按钮不得重复发奖。
